# 常见问题解答

## LLM基础问题

### Q1：LLM为什么会产生幻觉？

**原因**：模型只学会了"统计规律"，不理解"真假"

```
训练数据中有："地球是平的"（错误信息）
模型学到："地球"后面可能接"是平的"

模型不知道什么是真的，只知道什么"看起来像真的"
```

**具体原因**：
1. **训练数据偏差**：网络数据包含错误信息
2. **概率生成**：模型基于概率生成，不是基于事实
3. **缺乏事实核查**：没有内置的事实验证机制
4. **模式匹配**：模型学习的是模式，不是理解

**解决方案**：
```
1. RAG：提供准确的信息源
2. 微调：在高质量数据上训练
3. 后处理：事实核查和验证
4. 提示工程：要求模型提供来源
```

### Q2：如何选择LLM？

**根据需求选择**：

| 需求 | 推荐模型 | 原因 |
|------|----------|------|
| 最强能力 | GPT-4o、Claude 3.5 | 综合能力最强 |
| 开源可部署 | LLaMA 3、Qwen 2 | 可控，可本地部署 |
| 中文任务 | 通义千问、文心一言 | 中文优化好 |
| 代码任务 | DeepSeek-Coder | 代码能力强 |
| 低成本 | 小模型 + 量化 | 资源需求低 |
| 长上下文 | Gemini 1.5、Claude 3.5 | 支持长文本 |

**选择考虑因素**：
```
1. 成本：API调用 vs 本地部署
2. 隐私：数据是否需要本地处理
3. 延迟：实时性要求
4. 准确性：对错误容忍度
5. 领域：特定领域需求
```

### Q3：LLM的上下文窗口有什么用？

**定义**：模型能处理的最大token数量

**重要性**：
```
影响：模型能"记住"多少信息
限制：超过窗口的信息会被截断
成本：更长的上下文 = 更高的成本
```

**不同模型的上下文窗口**：
```
GPT-3.5：4K token（约3000字）
GPT-4：128K token（约10万字）
Claude 3.5：200K token（约15万字）
Gemini 1.5：1M token（约75万字）
LLaMA 3：8K token（约6000字）
```

**使用建议**：
```
1. 控制输入长度
2. 分块处理长文档
3. 使用摘要技术
4. 选择合适上下文的模型
```

## 技术问题

### Q4：什么是Token？和单词有什么区别？

**定义**：LLM处理的基本单位

**区别**：
```
单词：语言学单位，按空格分割
Token：模型处理单位，按分词器规则分割

示例：
  "unhappiness" = 1个单词，3个token（un, happiness）
  "I love AI" = 3个单词，4个token（I, love, ' ', AI）
```

**为什么用Token而不是单词**：
```
1. 统一处理：不同语言统一表示
2. 未知词处理：可以分解未知词
3. 效率：平衡词汇表大小和表示能力
```

### Q5：为什么LLM需要微调？

**原因**：
```
预训练：学习通用语言能力
微调：适应特定任务和领域

类比：
  预训练 = 大学毕业（通用知识）
  微调 = 职业培训（专业技能）
```

**微调的好处**：
```
1. 专业化：适应特定领域
2. 格式化：输出特定格式
3. 安全性：减少有害输出
4. 效率：小模型也能有好效果
```

### Q6：什么是幻觉？如何减少幻觉？

**幻觉定义**：模型生成看似合理但错误的内容

**类型**：
```
1. 事实性幻觉：编造事实
2. 逻辑性幻觉：逻辑错误
3. 来源幻觉：编造引用来源
```

**减少幻觉的方法**：
```
1. RAG：提供准确信息源
2. 微调：在高质量数据上训练
3. 提示工程：要求提供来源
4. 温度设置：降低温度减少随机性
5. 后处理：事实核查
```

## 使用问题

### Q7：如何提高LLM的回答质量？

**提示工程技巧**：
```
1. 清晰具体：明确任务和要求
2. 提供上下文：给出背景信息
3. 指定格式：要求特定输出格式
4. 思维链：要求逐步思考
5. 少样本：提供示例
```

**参数设置**：
```
温度：0.7-1.0（平衡创造性）
top_p：0.9-0.95（控制多样性）
max_tokens：足够长但不过长
```

**使用策略**：
```
1. 迭代优化：根据结果调整提示
2. 验证结果：对重要信息进行验证
3. 多次尝试：不同参数多次生成
4. 人工审核：重要内容人工检查
```

### Q8：LLM可以处理多长的文本？

**取决于模型的上下文窗口**：
```
短上下文模型：4K-8K token
中等上下文：32K-128K token
长上下文：200K-1M token
```

**处理长文本的方法**：
```
1. 分块处理：将长文本分成小块
2. 摘要技术：先摘要再处理
3. 检索技术：RAG检索相关部分
4. 选择模型：使用长上下文模型
```

### Q9：为什么同一个提示，每次回答都不同？

**原因**：模型的随机性

**随机性来源**：
```
1. 温度参数：控制随机程度
2. 采样方法：top-p、top-k采样
3. 概率分布：基于概率选择下一个token
```

**控制方法**：
```
温度=0：确定性生成（相同输入相同输出）
温度>0：随机生成（每次不同）
设置种子：固定随机种子
```

## 部署问题

### Q10：如何部署本地LLM？

**部署选项**：
```
1. 开源模型：LLaMA、Qwen、Mistral
2. 推理框架：vLLM、TGI、llama.cpp
3. 量化：减少资源需求
4. 硬件：GPU服务器或消费级GPU
```

**部署步骤**：
```
步骤1：选择模型和框架
步骤2：下载模型权重
步骤3：配置推理环境
步骤4：启动服务
步骤5：测试和优化
```

**资源需求**：
```
7B模型：约14GB显存（FP16）
70B模型：约140GB显存（FP16）
量化后：显存需求减少50-75%
```

### Q11：本地部署和API调用哪个好？

**对比**：
| 方面 | 本地部署 | API调用 |
|------|----------|---------|
| 成本 | 前期高，长期低 | 按使用付费 |
| 隐私 | 数据本地处理 | 数据发送到云端 |
| 延迟 | 取决于硬件 | 网络延迟 |
| 维护 | 需要自己维护 | 服务商维护 |
| 能力 | 受限于开源模型 | 可以使用最强模型 |

**选择建议**：
```
选择本地部署：
  - 数据隐私要求高
  - 使用量大，长期成本考虑
  - 需要定制化
  - 有技术团队

选择API调用：
  - 初期探索阶段
  - 使用量不大
  - 需要最强模型
  - 没有技术团队
```

### Q12：如何评估LLM的性能？

**评估维度**：
```
1. 准确性：回答的正确性
2. 相关性：是否回答了问题
3. 流畅性：语言是否自然
4. 一致性：多次回答是否一致
5. 安全性：是否有害内容
```

**评估方法**：
```
1. 人工评估：专家评估质量
2. 自动评估：使用评估指标
3. A/B测试：对比不同版本
4. 用户反馈：收集用户评价
```

**常用评估基准**：
```
通用能力：MMLU、HellaSwAG
推理能力：GSM8K、ARC
代码能力：HumanEval、MBPP
中文能力：C-Eval、CMMLU
```

## 伦理问题

### Q15：使用LLM有什么伦理考虑？

**主要问题**：
```
1. 偏见和歧视：训练数据中的偏见
2. 隐私问题：可能泄露私人信息
3. 版权问题：训练数据可能有版权
4. 滥用风险：恶意使用可能造成伤害
5. 环境影响：训练和推理的能耗
```

**负责任的使用**：
```
1. 验证重要信息
2. 注意偏见和歧视
3. 保护隐私数据
4. 遵守法律法规
5. 考虑社会影响
```

**现实情况**：

1. 自动化部分任务，不是完全取代
2. 提高工作效率，需要人类监督
3. 创造新的工作机会
4. 需要人类进行判断和决策
```

**未来趋势**：
```
1. 人机协作：人类和AI合作
2. 技能升级：需要新技能
