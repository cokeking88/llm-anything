# 微调基础

## 什么是模型微调？

### 一句话定义
**一句话**：在预训练模型基础上，用特定数据进一步训练，让它更擅长特定任务

### 类比理解
```
预训练 = 大学毕业（通用知识）
微调 = 职业培训（专业技能）

通用GPT → 医疗GPT（医疗数据微调）
通用GPT → 代码GPT（代码数据微调）
```

## 微调的重要性

### 1. 专业化能力
```
通用模型：能做很多事，但都不专
微调模型：专注于特定任务，效果更好

示例：
  通用模型：能写代码，但代码质量一般
  微调模型：专门写Python，代码质量高
```

### 2. 格式控制
```
通用模型：输出格式不固定
微调模型：输出特定格式

示例：
  通用模型：回答可能用段落、列表等
  微调模型：总是用JSON格式输出
```

### 3. 安全性提升
```
通用模型：可能生成有害内容
微调模型：更安全，更符合伦理

示例：
  通用模型：可能回答敏感问题
  微调模型：礼貌拒绝敏感请求
```

### 4. 效率提升
```
通用模型：需要复杂Prompt
微调模型：简单指令即可

示例：
  通用模型：需要详细的角色扮演Prompt
  微调模型：简单说"总结这篇文章"即可
```

## 微调方法对比

### 全参数微调
```
方法：更新所有模型参数
优点：效果最好
缺点：需要大量GPU和显存
适用：资源充足，追求最佳效果
```

### LoRA（低秩适配）
```
方法：只训练低秩矩阵
优点：参数少，显存需求低
缺点：效果略低于全参数微调
适用：通用场景，资源有限
```

### QLoRA（量化LoRA）
```
方法：4-bit量化 + LoRA
优点：显存需求极低
缺点：训练速度较慢
适用：显存有限，如单张消费级GPU
```

### Adapter（适配器）
```
方法：在模型中插入小网络
优点：可插拔，多任务
缺点：推理速度受影响
适用：多任务场景
```

### 方法对比表
| 方法 | 参数量 | 显存需求 | 效果 | 适用场景 |
|------|--------|----------|------|----------|
| 全参数微调 | 100% | 极高 | 最好 | 资源充足 |
| LoRA | 0.1%-1% | 低 | 好 | 通用场景 |
| QLoRA | 0.1%-1% | 更低 | 好 | 显存有限 |
| Adapter | 1%-5% | 中 | 中等 | 多任务 |

## 预训练 vs 微调

### 预训练
```
目标：学习通用语言表示
数据：海量无标注数据（TB级）
任务：语言建模
计算：需要大量GPU，训练数周到数月
结果：通用基础模型
```

### 微调
```
目标：适应特定任务
数据：少量有标注数据（MB到GB级）
任务：分类、生成等具体任务
计算：需要少量GPU，训练数小时到数天
结果：任务专用模型
```

### 关系
```
预训练 + 微调 = 完整学习过程
预训练提供基础能力
微调提供专业技能

类比：
  预训练 = 基础教育（学习语言、数学等）
  微调 = 职业培训（学习专业技能）
```

## 微调类型

### 指令微调（Instruction Tuning）
```
数据：指令-回答对
目标：让模型学会遵循指令
示例：
  指令："请总结这篇文章"
  回答：[文章摘要]
```

### 对话微调（Chat Tuning）
```
数据：多轮对话
目标：让模型学会对话
示例：
  用户：你好
  助手：你好！有什么可以帮你的吗？
  用户：今天天气怎么样？
  助手：[天气信息]
```

### 领域微调（Domain Tuning）
```
数据：领域特定数据
目标：让模型擅长特定领域
示例：
  医疗领域：医学文献、病例
  法律领域：法律条文、案例
  代码领域：开源代码、文档
```

### 风格微调（Style Tuning）
```
数据：特定风格的文本
目标：让模型学会特定风格
示例：
  正式风格：商务邮件、报告
  创意风格：诗歌、故事
  技术风格：技术文档、论文
```

## 微调流程

### 完整流程
```
1. 准备数据
   ↓
2. 加载模型和Tokenizer
   ↓
3. 配置微调方法（LoRA/QLoRA）
   ↓
4. 设置训练参数
   ↓
5. 训练
   ↓
6. 保存模型
   ↓
7. 测试和部署
```

### 关键步骤详解

#### 1. 数据准备
```
数据格式：指令-回答对
数据清洗：去除低质量数据
数据增强：增加数据多样性
数据划分：训练集、验证集、测试集
```

#### 2. 模型加载
```
选择基础模型：如LLaMA、Qwen
加载Tokenizer：处理文本
加载模型权重：预训练权重
```

#### 3. 配置微调
```
选择方法：LoRA、QLoRA等
设置参数：秩、alpha等
选择目标层：哪些层应用微调
```

#### 4. 训练配置
```
学习率：通常1e-4到5e-4
批次大小：根据显存调整
训练轮数：通常3-10轮
优化器：AdamW
```

#### 5. 训练执行
```
前向传播：计算损失
反向传播：计算梯度
参数更新：优化器更新
监控：损失曲线、评估指标
```

#### 6. 模型保存
```
保存LoRA权重：只保存训练的部分
合并权重：可选，合并到原模型
格式转换：转换为推理格式
```

## 微调挑战

### 1. 数据质量
```
挑战：垃圾进，垃圾出
解决：数据清洗、质量控制
平衡：数量 vs 质量
```

### 2. 过拟合
```
挑战：在训练数据上表现好，新数据表现差
解决：早停、正则化、数据增强
监控：验证集性能
```

### 3. 灾难性遗忘
```
挑战：微调后忘记预训练知识
解决：保留一部分预训练数据、使用LoRA
评估：通用能力测试
```

### 4. 计算资源
```
挑战：需要GPU和显存
解决：使用LoRA/QLoRA、混合精度训练
优化：梯度累积、梯度检查点
```

## 微调评估

### 评估维度
```
1. 准确性：任务完成质量
2. 相关性：是否符合要求
3. 流畅性：语言是否自然
4. 安全性：是否有害内容
5. 遗忘程度：通用能力保留
```

### 评估方法
```
1. 定量评估：指标计算（准确率、BLEU等）
2. 定性评估：人工评估
3. 对比评估：与基础模型对比
4. 应用评估：实际使用效果
```

## 最佳实践

### 1. 数据质量优先
```
原则：质量比数量重要
做法：精心设计数据，人工审核
```

### 2. 小步快跑
```
原则：先小规模测试，再大规模训练
做法：先用少量数据测试，确认有效后再扩大
```

### 3. 监控和调整
```
原则：持续监控，及时调整
做法：监控损失曲线，调整超参数
```

### 4. 全面评估
```
原则：多维度评估
做法：不仅评估目标任务，也评估通用能力
```
